Work management as a nostr convention

One work queue for your team and your AI agents.

rd is a single command-line tool that humans and agents drive the same way — create, claim, delegate, close. Why rd

A work tracker that isn't a web app

Issue trackers assume a server, an account, and a browser. rd assumes a terminal and a signing key. That one change — work as signed events in a log you own, instead of rows in someone's database — is what lets agents and humans share it, work offline, and keep receipts forever.

🤝

Humans and agents, one CLI

There's no separate agent API and no MCP server to run. A person at a terminal and a Claude Code session run the exact same commands — rd create, rd claim, rd done. Output is pipe-friendly: bare IDs when piped, tables when interactive.

📦

One binary, no backend

No Postgres, no Redis, no SaaS, no login. Install one self-contained binary and run rd init. A secp256k1 signing key is minted on first use — no separate identity ceremony.

📴

Local-first, works offline

Your append-only log (.ready/nostr-log.jsonl) is the authority. Relays only sync it between machines. Pull the network cable and every command still works — writes buffer and flush when a relay reappears.

🔏

A signed audit trail, forever

Every create, claim, status change, delegation, and gate is a signed nostr event, appended once and kept. Who did what, when, and why — cryptographically attributed, never compacted, never archived away.

🔐

Confidential by default

Titles, descriptions, and close reasons are encrypted so only granted members can read them — a relay operator sees opaque ciphertext. The routing graph stays queryable. --public opts out.

🔀

Built for delegation

Every item carries for (who needs the outcome) and by (who's doing it). Agents escalate decisions to humans with rd gate; humans rule with rd approve. Delegation is an act you record, not an assumption.

How it works

Create → Claim → Close

Every work item follows the same three-step lifecycle, and the same commands serve a human at a keyboard or an agent reading a prompt. Priority sets the ETA — P0 is now, P3 is 72 hours.

# Create a work item — prints the item ID when piped rd create "Ship login page" --priority p1 --type task myproject-45e # What needs attention right now? rd ready myproject-45e p1 inbox 3h Ship login page # Claim it — you're now the performer rd claim myproject-45e claimed myproject-45e # Close with a reason — the audit trail records the "why" rd done myproject-45e --reason "Login page ships with JWT auth" closed myproject-45e (done)

The attention engine

Named views filter the log by who you are and what's actionable. rd ready is the default lens; the rest slice by status, ownership, and time.

# Views: ready, work, pending, overdue, delegated, my-work rd ready --view overdue rd list --status active --priority p0,p1 rd list --by atlas/worker-3 --all --json

Delegate, block, escalate

Reassign the performer, wire a dependency (even across projects you own), or push a decision to a human without leaving the terminal.

rd delegate api-a1b --to atlas/worker-3 rd dep add api-a1b api-9f2 rd gate api-a1b --gate-type design \ --description "PKCE vs device flow?"

Playbooks for recurring shapes

Incident runbooks, release prep, feature rollouts — stamp a template into real items with their dependencies already wired, instead of decomposing by hand every time.

rd playbook list rd engage release-checklist --project api engaged release-checklist → 6 items (deps wired)

Any nostr client can read it

Because every operation is a plain signed event, the log is the integration surface. Point any nostr-aware tool at a relay and read the boards, cards, and status events directly — no rd-specific API required.

Command reference

rd delegate <id> --to <identity>Assign to a person, agent, or automaton
rd done <id> --reason "..."Close with a reason — recorded to the audit trail

For agents

Built for the ones doing the work

rd's primary users are AI agents; humans use it too. Drop one of these into your project and Claude Code starts tracking work immediately — no MCP server, no special integration, just rd on PATH.

1 · Add it to CLAUDE.md

Paste this into your project's CLAUDE.md. Claude reads it at session start and uses rd like any other CLI — including recovering its place after a context reset.

## Work Management rd is on PATH and auto-detects the project from the working directory. - Start every session: `rd ready` (what's actionable now) - Claim before working: `rd claim <id>` - Close with a reason: `rd done <id> --reason "..."` - After a context reset: `rd ready --view work`, then `rd show <id>` to reload - New items: `ITEM=$(rd create "..." --type task --priority p1)` - Recurring shapes: `rd playbook list` before decomposing by hand - Everything supports `--json` for structured output

2 · Or drop in the skill file

Download SKILL.md into .claude/skills/rd/. Claude Code discovers it automatically and gains full rd command knowledge.

mkdir -p .claude/skills/rd curl -fsSL https://ready.3dl.dev/SKILL.md -o .claude/skills/rd/SKILL.md

3 · Run a multi-agent swarm on one queue

Spawn parallel Claude Code agents that share a work queue. Each gets its own signing identity; the filesystem handles isolation — no env-var juggling.

# Owner seeds the queue rd create "Build auth" --priority p0 --type task rd create "Build API" --priority p1 --type task rd create "Write tests" --priority p1 --type task # Each agent joins from its own worktree, then pulls from the shared queue rd join $(rd invite) ITEM=$(rd ready | head -1) rd claim $ITEM && rd done $ITEM --reason "Auth module complete"

4 · Let an agent escalate to a human

The agent hits a decision it shouldn't make alone. It posts a gate; the human rules from any terminal. No dashboard, no ticket system — just signed events.

# Agent: "I need a ruling." rd gate myapp-d4e --gate-type design \ --description "PKCE vs device flow for mobile auth" {"id":"myapp-d4e","gate_type":"design","status":"waiting"} # Human: sees it, decides, moves on rd gates myapp-d4e p1 design PKCE vs device flow rd approve myapp-d4e --reason "PKCE. Need the browser redirect." # Agent's item is back to active — it continues from there.

Zero-config identity per actor

Every actor signs with its own secp256k1 key under $RD_HOME. $RD_ACTOR selects which key signs, so the owner and a named agent on the same host are attributed distinctly — RD_ACTOR=agent:pm rd claim <id> and the transition is stamped to that key.

The same commands, always

There is no separate agent surface to learn or keep in sync. Whatever a human can do, an agent does with the identical command — which is exactly why a person and a swarm can share one board without stepping on each other.

Confidential boards

Encrypt the work that has to stay private

On a confidential board (the default), a work item's free text is encrypted so only granted members can read it — while the routing graph stays clear enough that sync, dedupe, and every query keep working untouched. Confidentiality is a property of the log, not a feature of a server you have to trust.

What's sealed, what stays clear

Encrypted into the event body: the title, description, close reason, and waiting-on. Left clear so the relay can still index and filter: status, dependencies, priority, assignees, parent, due date. Labels are HMAC-tokenized — filterable by members, opaque to everyone else.

# What a non-member sees when they REQ the relay: kind: 30302 (item card) d: api-d4e ← clear (routing) status: active ← clear priority: p1 ← clear l: 7f3a9c… ← tokenized label content: kJ4d…b2== ← sealed: title + body

Keys live in the log, not on a server

The per-board key is wrapped to each member (and to the owner) inside an owner-signed grant, using NIP-44. So the key is recoverable from the replicated log by anyone holding the identity key — nothing extra to back up, and a relay operator never sees plaintext or keys.

Forward secrecy on revoke rd revoke mints a new key epoch and re-wraps it to the remaining members. A removed member keeps what it already read but can't decrypt anything authored after — and its own historical reads still resolve.
✅

On by default

rd init creates a confidential board; rd init --public opts out. Granted members read plaintext transparently — no manual key handling, ever.

🔑

Owner is the root of trust

Only the board owner mints and distributes the key, inside the same signed grant that confers write access. Membership is the read key — one action, not two.

🔒

Real crypto, standard NIPs

ChaCha20-Poly1305 over the content, NIP-44 v2 to wrap the key, validated against the official NIP-44 test vectors. The relay is a dumb, replaceable cache of ciphertext.

Nostr-native, no server

Signed events, not a database

Ready isn't a project-management app with a backend bolted on. rd is a convention — a structured vocabulary of work operations that maps onto ordinary signed nostr events any relay or client can read. That's the whole architecture.

State is a projection of the log

When you run rd create, rd builds a signed item card (kind 30302) and a NIP-34 status event, appends both to your local append-only log, and publishes them to your relays. rd claim appends another status event with a causal link back to the card. An item's current state is computed by replaying the log — there is no separate database to drift out of sync.

The core work vocabulary

work:create new item with full payload work:claim accept work, become performer work:status transition status work:delegate assign / reassign performer work:block wire a dependency work:gate request human escalation work:gate-resolve resolve a gate work:grant owner-rooted role grant work:update modify fields work:close terminal close with reason

Identity is a key, not an account

Every actor is a secp256k1 keypair under $RD_HOME — minted on first use, no signup. The same key Schnorr-signs your events and does the ECDH that unwraps confidential board keys. Membership and trust are owner-rooted role grants (kind 39301): revocable, graded, and derivable from the signed log alone.

Relays are replaceable caches

A relay's only job is to move signed events between machines. Run your own (a self-hosted write-allowlist admits only owner-granted keys), rent a hosted one, or point at several for redundancy. Lose them all and your local log is intact — work never lived on the relay.

Federation without a mothership

Each project is its own nostr board (kind 30301) with its own authoritative local log. Relays sync boards between machines; there is no central server that could go down or hold your data hostage.

Relays (self-hosted, 2+ for no single point of failure) ├ write-allowlist: owner-rooted role grants ├ Board: api (local log authoritative) ├ Board: frontend (local log authoritative) └ Board: infra (local log authoritative)

Nothing to compact — ever

The append-only log is the audit trail. Every create, claim, status change, delegation, gate, and grant is a signed event, appended once and kept.

The trail survives Relays may drop old data — your local log never does. Nothing is archived away; nothing needs to be.

Built on open nostr standards

Ready layers a work vocabulary on public NIPs — addressable boards and cards, NIP-34 status events, NIP-22 comments, NIP-44 v2 encryption, NIP-77 Negentropy sync. There's no custom wire protocol to implement: any nostr client already speaks the transport, and the events are the API.

The rd CLI — event schema, projection engine, confidential envelope, and named views. MIT-friendly, all in the open.
